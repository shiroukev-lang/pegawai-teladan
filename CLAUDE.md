## Running the App

Two processes must run simultaneously:

**API server** (MySQL backend):
```bash
cd src/local-setup/server
node index.js          # production
npm run dev            # with auto-reload via nodemon
```

**Frontend** (Vite dev server):
```bash
npm run dev            # starts on port 3000
npm run build          # outputs to build/
```

Prerequisites: Laragon must be running with MySQL active. The database `tlhp_db` is initialized via `src/local-setup/database/schema.mysql.sql`.

## Architecture

This is a **two-phase employee voting app** (Pemilihan Pegawai Teladan) for Inspektorat Jenderal. It was originally generated by Figma Make and exported as a Supabase project, then adapted to run locally with MySQL.

### Frontend (React + Vite + TypeScript)

Routing in `src/App.tsx`:
- `/admin` → `AdminDashboard` (default redirect)
- `/vote?key=...` → `ValidationPage` (validates employee NIP + unique link)
- `/phase1` → `Phase1Voting` (star ratings 1–5 for each candidate per criteria)
- `/phase1/results` → `Phase1Results`
- `/phase2` → `Phase2Voting` (pick 1 best candidate per criteria)
- `/phase2/results` → `Phase2Results`

All API calls go through `src/utils/api-config.ts`:
```ts
getApiUrl('/endpoint')   // builds http://localhost:3001/make-server-ea54a030/endpoint
getHeaders()             // returns { 'Content-Type': 'application/json' }
```
The prefix `/make-server-ea54a030` is kept to match the original Supabase Edge Function name. `VITE_API_URL` env var overrides the base URL.

### API Server (`src/local-setup/server/index.js`)

Express.js using `mysql2/promise` with a connection pool. All routes are mounted under `/make-server-ea54a030`. Credentials from `.env`:
- `DB_HOST=localhost`, `DB_PORT=3306`, `DB_USER=root`, `DB_PASSWORD=`, `DB_NAME=tlhp_db`

### Database (MySQL 8.4 via Laragon)

Schema in `src/local-setup/database/schema.mysql.sql`. Tables:
- `app_config` — key/value store for `current_phase` (1/2/3), `bagian` JSON list, `initialized` flag
- `pegawai` — voters (NIP, nama, bagian, unique_key, has_voted_phase1/2)
- `kandidat` — candidates (max 5 per bagian enforced by trigger)
- `vote_phase1` — star ratings (1–5) per voter per candidate per 7 criteria
- `vote_phase2` — one candidate choice per 7 criteria per voter

Three views used by the API: `view_phase1_results`, `view_phase1_winners` (ROW_NUMBER OVER PARTITION BY bagian), `view_phase2_results`.

### Voting Flow

1. Admin creates pegawai records → generates unique voting links (`/vote?key=...`)
2. Admin adds kandidat (up to 5 per bagian/department)
3. Admin sets `current_phase = 1`
4. Each pegawai visits their link, enters NIP → validated, then votes Phase 1 (star ratings for all candidates in their bagian)
5. Admin sets `current_phase = 2`; Phase 1 winners (1 per bagian) become Phase 2 candidates
6. Each pegawai votes Phase 2 (1 best per criteria across all bagian winners)
7. Admin views final winner

### Authentication

Admin dashboard uses hardcoded passwords checked client-side in `AdminPasswordDialog.tsx`:
- Super admin (`tlhpteladan123`): full access including pegawai/kandidat management and settings
- Admin (`adminteladan123`): read-only overview and results tabs

Session is stored in `sessionStorage`. `logoutAdmin()` is exported from the dialog component.

### Key Data Shapes

The API always returns boolean `hasVotedPhase1`/`hasVotedPhase2` (converted from MySQL TINYINT via `!!`). Phase 1 vote body: `{ nip, votes: { [kandidatNip]: { 'Berorientasi Pelayanan': 1-5, 'Akuntabel': 1-5, ... } } }`. Phase 2 vote body: `{ nip, votes: { 'Berorientasi Pelayanan': kandidatNip, ... } }`.

The 7 fixed criteria: Berorientasi Pelayanan, Akuntabel, Kompeten, Harmonis, Loyal, Adaptif, Kolaboratif.

The 6 fixed bagian: Sekretariat Inspektorat Jenderal, Inspektorat Wilayah I–V.
